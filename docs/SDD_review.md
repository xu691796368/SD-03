
# SD-03 分布式缓存系统 - SDD 过程回顾

## 1. 项目概述

SD-03是围绕分布式缓存核心原理与 TCP 网络编程技术落地的轻量级实践项目。整个项目聚焦五大核心模块的落地：LRU 缓存淘汰、一致性哈希分片、自定义二进制协议、TCP 并发服务器，以及简化版主从复制，核心目标是通过实战吃透分布式缓存的底层逻辑。

---

## 2. SDD 全流程回顾

### 2.1 文档驱动开发流程

这次项目严格遵循SDD原则，每个阶段都用文档设定目标和标准，避免开发过程中需求漂移：
先是通过 proposal.md 明确 “做什么” 和 “为什么做”，再用 specs.md 定义场景级的验收标准（GIVEN-WHEN-THEN 格式），接着通过 design.md 落地 “怎么做”（包括架构、接口和数据结构设计），然后拆分成 tasks.md 的原子任务，再按任务粒度完成代码实现和测试，最后通过测试结果和复盘文档验证、积累ai coding经验。

### 2.2 各阶段的收获

#### Phase 1: 基础架构（Task 1.1-1.6）
这一阶段的核心收获是：以协议层为切入点，先把客户端和服务端的通信契约定义清楚，再去实现业务逻辑，后续模块集成时的风险直接降了一大截。

#### Phase 2: 核心功能（Task 2.1-2.7）
严格按 “cache → shard → node → server → replication” 的依赖顺序开发，每层只依赖下层，整个架构始终保持清晰，不会越写越乱。

#### Phase 3: 测试与优化（Task 3.1-3.11）
测试客户端采用嵌入式集群（EmbeddedCluster）实现测试隔离 —— 每个测试用例都创建独立的临时集群，彻底避免不同用例之间的状态干扰，测试结果的稳定性大幅提升。

## 3. 遇到的挑战与解决方案

### 3.1 二进制协议编解码的边界处理

开发初期，在二进制协议编解码环节踩了坑。协议帧格式定义为Command(1B) + KeyLen(4B) + ValueLen(4B) + Key + Value，但实际处理时遇到了几个关键问题：
- 响应帧中 Status 字节的定位（最初错误地从位置 9 读取，应从位置 10）
- nil Key 和超长 Key/Value 的防御性校验
- 帧头声明长度与实际数据长度不匹配的检测

针对这些问题，解决方案如下：
- 使用大端字节序（Big-Endian）统一编码
- 在 EncodeRequest 阶段前置参数校验，拒绝非法输入
- 在 DecodeRequest 阶段增加长度一致性检查
- 引入 FrameError 类型和 GetErrorCode 辅助函数，实现错误码的程序化处理

二进制协议的每一位偏移都必须精确定义并写进文档。差一个字节，整个链路的数据都会错乱。

### 3.2 LRU 缓存的并发安全与淘汰准确性

LRU 缓存是核心模块，但多客户端并发操作时，需要注意：
- 多客户端并发 SET/GET 时需要保证缓存数据一致性
- LRU 淘汰必须精确移除"最久未使用"的条目，而非"最早写入"的条目

解决方案如下：
- 使用 `sync.RWMutex` 保护缓存读写 —— 读操作加共享锁，写操作加独占锁，兼顾并发性能和数据安全；
- 严格区分"写入时间"与"最后访问时间"：GET操作也触发位置提升

为了验证逻辑正确性，设计了 C06（容量满淘汰）、C07（热点数据保留）、C08（删除释放空间）三个边界测试用例，把淘汰逻辑的细节都验证到位了。

### 3.3 一致性哈希的数据分布均衡

一致性哈希的核心是让数据均匀分布在节点上，但实际落地时遇到了三个问题：
- 3 个物理节点路由 1000 个 Key 时，数据分布偏差容易超过 30%；
- 虚拟节点数量不好选 —— 太少会导致分布不均，太多又会增加内存开销；
- 节点增减时，希望数据迁移量尽可能小（这也是一致性哈希的核心价值）

解决方案如下：
- 给每个物理节点分配 100 个虚拟节点（总共 300 个），平衡分布精度和内存开销
- FNV-1a 对简单字符串 "NodeA#0" 的哈希分布不够均匀。需要改进虚拟节点的哈希生成策略，使用两阶段 FNV-1a 哈希来提升分布质量
- 定义偏差计算公式：`偏差 = abs(单节点数量 - 均值) / 均值` ，量化验证分布效果；

通过 S06 测试验证，1000 个 Key 在 3 个节点的分布偏差都控制在 30% 以内，达到了预期目标。这里的经验是：虚拟节点是分布式系统负载均衡的基石，用 “少量内存开销” 换 “数据分布均匀性”，性价比很高。

### 3.4 TCP 服务器的并发连接管理

TCP 服务器需要支持多客户端并发连接，但初期遇到了几个稳定性问题：
- 需要同时支持10+客户端并发连接，容易出现数据串扰；
- 客户端异常断开后，服务器可能崩溃，且资源清理不彻底；

解决方案如下：
- 每个客户端连接使用独立的 goroutine 处理（`go handleConnection(conn)`），隔离并发请求；
- 使用随机端口（:0）创建测试集群，避免端口冲突
- 通过 channel + WaitGroup 收集并发测试中的错误，统一汇总和排查

通过 T08（5 客户端并发）、T09（10 客户端压力测试）验证，服务器能稳定处理 50-400 次并发操作，没有出现崩溃或数据串扰问题。

### 3.5 测试隔离与可重复性

测试阶段遇到的问题是 “测试用例相互干扰” —— 不同用例共享集群时，前一个用例的状态会影响后一个的结果，导致测试结果不稳定。

解决方案如下：
- 每个需要集群的测试用例，都调用 `tempCluster()` 创建独立的临时集群
- 使用随机端口绑定，避免端口冲突
- 测试结束后通过 `defer stopCluster()` 确保资源释放
- 并发测试中引入适当延时（如 100ms）等待服务器处理异步事件

这么做之后，所有测试用例都能独立运行，结果稳定可重复，不用再花时间排查 “偶发失败” 的问题。

## 4. AI使用心得

这次项目使用AI贯穿了开发全流程，包括：
（1）需求分析：proposal/specs文档生成与优化	
（2）架构设计：design.md模块划分与接口定义	
（3）任务分解：tasks.md任务拆分
（4）代码实现：核心算法实现（LRU / 哈希 / 协议）	
（5）测试编写：单元测试 / 集成测试 / 边界用例
（6）调试排障：并发问题 / 协议偏移 / 端口冲突
（7）文档优化：注释规范 / README / 文档对齐

### 4.1 AI配合SSD模式
1、不是直接用AI生成的文档初稿，而是多轮精炼；每轮只盯一个维度（完整性→准确性→一致性→格式），既能保证效率，也能避免AI生成的内容 “大而空”。
2、以Spec场景为输入，让AI帮我们落地测试用例；这种模式的核心是：明确的GIVEN-WHEN-THEN结构能让AI精准理解验收标准，生成的测试用例命中率很高。
3、每个Task完成后，用AI辅助验证；不用等所有功能写完再补测试，也能及时发现问题。

### 4.2 AI使用的注意事项
1、必须验证生成代码：AI生成的代码不一定能编译通过，更不一定符合业务逻辑，必须跑通测试才能用
2、关注上下文窗口：复杂任务要分阶段处理，比如拆分 “协议编解码” 为 “类型定义”“编码函数”“解码函数” 三个子任务，避免AI上下文丢失导致内容不一致
3、保持自我判断：架构决策、技术选型必须由自己主导，AI 只作为方案参考 —— 比如 LRU 实现选标准库，是基于项目目标的决策，而非直接用 AI 的建议
4、文档与代码同步：AI修改代码后，一定要同步检查相关文档，避免 “代码改了，文档还是旧的”
5、提供具体的上下文：给AI的提示词中，最好明确标注 “基于spec第X节场景”“参考 tests/integration 代码”，输出质量会大幅提升，不然消耗的token也会很大（特别是项目文件越来越多的情况）
6、使用精确指令：避免模糊的 “帮我写个测试”，而是 “帮我写一个测试 LRU 缓存容量满时淘汰最久未使用条目的单元测试”
